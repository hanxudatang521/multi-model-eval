# 模型效果对比工具产品需求说明

本文档用于沉淀产品讨论中已经确认的需求、交互结论和开发约束。  
后续本对话主要用于讨论产品需求；一旦形成明确结论，需要同步更新本文档，方便另一个开发对话框里的智能体直接读取。

## 产品定位

这是一个面向产品经理的模型效果评测工具。

核心场景是：产品经理上传一份开发给出的原始评测表，对同一 query 下两个模型或两个版本的输出进行对比、标注和导出反馈，最终把评测结论反馈给开发。

产品不做营销展示，重点是高频评测效率、可读性、标注准确性和导出给开发的可用性。

## 当前运行形态

- 纯静态网页。
- 支持本地访问，也支持 GitHub Pages 线上访问。
- 本地服务推荐端口：`4173`。
- 本地访问地址：`http://127.0.0.1:4173/index.html`。

## 当前项目文件

- `index.html`：页面结构。
- `styles.css`：页面样式。
- `app.js`：文件解析、Markdown 渲染、评测交互、导出逻辑。

- `产品需求说明.md`：产品需求沉淀文档，也就是本文档。

## 已确认的数据输入规则

支持上传：

- `.csv`
- `.xlsx`

第一行必须是表头。

### 标准评测表结构

推荐包含：

- `query` 列。
- 两个模型输出列，列名可包含“大模型回复”“模型回复”“response”“output”等关键词。
- 其他开发参数列，例如 `label`、`tool_call_name`、`tool_call_args`、`工具结果`、`搜索结果` 等。

### 导入识别规则

- 优先识别表头为 `query` 的列作为 query。
- 优先识别列名包含“大模型回复 / 模型回复 / response / output”等关键词的两列作为左右模型输出。
- 评测页面只展示 query 和两个模型输出。
- 其他原始参数列不在评测页面展示，但必须在评分导出时保留。
- 上传新文件后，需要清空当前所有整体评分和局部反馈缓存，避免旧数据污染新评测。

## 已确认的页面布局

### 左侧区域

左侧用于操作 query 和文件：

- 上传 CSV / XLSX。
- 展示数据格式提示。
- 搜索 query。
- 打乱 query 顺序。
- 下载当前 query 的对比图。
- 导出评分。
- 查看总评分。
- 上一条 / 下一条。
- query 下拉选择。
- query 列表。

### 右侧区域

右侧用于评测当前 query：

- 顶部展示当前 query。
- 展示当前 query 在两个模型下的输出。
- 支持 Markdown 渲染。
- 支持容器样式切换：
  - Web 端页面。
  - 移动端页面。

## 已确认的容器样式

### Web 端页面

默认容器。

左右模型输出以普通网页卡片并排展示，卡片内支持 Markdown 格式。

### 移动端页面

使用 `设备外壳.png` 作为设备外壳。

两个模型输出分别放入两个移动端设备壳中。手机内内容区域独立滚动，用于模拟移动端产品页面中的阅读体验。

后续如果产品经理补充移动端真实页面结构，需要继续调整手机内布局。

## 已确认的整体评分功能

当前已有整体评分能力：

- 左好。
- 相同。
- 右好。

规则：

- 首次对某 query 评分后，自动跳转到下一个未评分 query。
- 修改已有评分时，不自动跳转。
- 自动跳转后，右侧内容回到顶部。
- 左侧 query 列表需要展示该 query 的评分标记。
- 上传新文件后，清空评分缓存。

后续方向：

- 局部反馈能力上线后，整体评分可以保留、弱化或未来再讨论是否移除。

## 已确认的总评分功能

总评分以当前页面浮层展示，不跳转新页面。

总评分重点展示：

- 左好 / 相同 / 右好的评分占比。
- 评分数。
- 左右模型胜出差异。

不展示每个 query 的逐条明细。

## 已确认的对比图导出功能

支持一键下载当前 query 的两个模型对比 PNG。

要求：

- query 标题最多展示 20 个字。
- 图片中支持 Markdown 基础格式，而不是原样展示 `##`、`**` 等 Markdown 符号。
- 支持标题、加粗、列表、引用、代码块、表格等常见 Markdown 格式。

## 已确认的评分导出功能

导出格式为 `.xlsx`。

导出目标是反馈给开发，因此必须保留开发原始表格中的所有参数列，不允许只导出简化后的 query 和模型结果。

### Sheet 1：原始数据+评分

保留上传文件中的所有原始列，并在后面追加：

- 评分结果。
- 胜出模型。
- 左侧模型列名。
- 右侧模型列名。
- 是否已评分。
- 左侧局部好评数。
- 左侧局部差评数。
- 右侧局部好评数。
- 右侧局部差评数。
- 局部反馈总数。
- 左侧好的具体反馈。
- 左侧坏的具体反馈。
- 右侧好的具体反馈。
- 右侧坏的具体反馈。

局部具体反馈汇总格式：

```text
1. 标记内容：xxx
具体原因：xxx

2. 标记内容：xxx
具体原因：xxx
```

### Sheet 2：局部反馈明细

每一条划词反馈导出为一行。

字段：

- query。
- 模型名称。
- 模型列名。
- 标记内容。
- 评价。
- 好的具体原因。
- 坏的具体原因。
- 前文。
- 后文。
- 文本起始位置。
- 文本结束位置。
- 原始行号。
- 创建时间。

## 已确认的局部反馈功能

局部反馈用于解决“整体好坏评分不够精确”的问题。

产品经理可能面对 1800 字左右的长文本，需要指出某个标题、某一段、某一句具体好或坏，并填写原因。

### 交互流程

1. 用户在某个模型结果内容区内划选一段文字。
2. 松开鼠标后，在选区附近出现 `好 / 坏` 浮层按钮。
3. 用户点击 `好` 或 `坏`。
4. 弹出原因输入框。
5. 用户填写具体原因并确认。
6. 系统保存这一条局部反馈。
7. 被标记的片段在正文中高亮。

### 多条反馈

同一个 query、同一个模型结果中，允许多次划选、多条反馈。

例如同一段 1800 字内容中，可以有：

- 第 1 条：标题好，原因是结构清晰。
- 第 2 条：某段坏，原因是事实不准确。
- 第 3 条：某句坏，原因是表达太笼统。
- 第 4 条：结尾好，原因是给出明确建议。

这些反馈都挂在同一个 query 和同一个模型结果下。

### 局部反馈展示

- 被反馈过的片段需要高亮。
- 好：绿色高亮。
- 坏：黄色高亮。
- 模型标题附近显示 `局部反馈 N`。
- 点击 `局部反馈 N` 可以查看该模型结果下的反馈列表。
- 列表里可以看到标记内容、好/坏、原因、创建时间。
- 支持删除单条局部反馈。

### 局部反馈定位策略

当前原型采用“渲染后纯文本 offset + 上下文”的方案。

原因：

- 页面展示的是 Markdown 渲染后的 HTML。
- 原始 Markdown 文本和展示文本并非严格一一对应。
- 第一版优先满足产品经理评测和导出反馈，不强求回写原始 Markdown AST。

每条局部反馈记录：

- `selectedText`
- `startOffset`
- `endOffset`
- `beforeContext`
- `afterContext`
- `reason`
- `type`
- `createdAt`

后续如果需要自动回写训练数据或精确定位原始 Markdown，可再升级为 Markdown AST 映射方案。

## 已确认的 AI 辅助评测方向

当前阶段不直接在前端调用大模型 API。

原因：

- 当前工具是纯静态页面，直接在前端放 API Key 有泄露风险。
- 批量自动评测会涉及成本、限流、失败重试和进度管理。
- 不同大模型 API 形式不同，过早接入会增加复杂度。

因此第一版采用轻量半自动方案：

```text
生成当前 query 的 AI 评测 Prompt
```

### 第一版功能：生成当前 query 的 AI 评测 Prompt

工具根据当前 query 生成可复制给大模型的评测 Prompt。

Prompt 需要包含：

- 评测任务说明。
- 输出格式要求。
- 当前 query。
- 左侧模型名称和模型回复。
- 右侧模型名称和模型回复。
- 需要大模型判断：左好 / 相同 / 右好。
- 需要大模型说明判断理由。
- 需要大模型分别指出左侧优点、左侧问题、右侧优点、右侧问题。
- 输出要便于产品经理阅读和判断，不要求回填工具。

第一版不支持 AI 评测结果导入。

产品经理在外部大模型里查看 AI 评测结果后，回到本工具中自行手动完成：

- 整体评分：左好 / 相同 / 右好。
- 或局部反馈：划选片段并填写好/坏原因。

AI 评测结果本身暂不进入导出的 Excel。导出仍以用户手动评分和用户手动局部反馈为准。

### 后续可能升级方向

后续如果验证 AI 评测有价值，再考虑增加后端服务，实现真正的自动评测。

后端服务需要负责：

- 安全保存 API Key。
- 调用多个大模型。
- 控制并发。
- 处理失败重试。
- 记录成本。
- 返回结构化评测结果。

## 当前待继续讨论

- 局部反馈是否需要支持编辑原因。
- 点击高亮片段时，是否要直接打开该条反馈详情。
- 是否需要支持“定位到原文”的滚动跳转。
- 是否需要隐藏或移除底部整体评分按钮。
- 移动端设备壳内是否要模拟更真实的产品页面结构。
- 多模型超过两个时的评测和导出规则是否需要重新设计。
- AI 评测 Prompt 是否需要同时支持“复制到剪贴板”和“下载 txt”。
- AI 评测 Prompt 具体模板文案如何设计。
